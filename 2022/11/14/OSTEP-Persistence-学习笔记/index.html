#38 Redundant Arrays of Inexpensive Disks(RAIDs) - CRUX : How to make a large , fast , reliable disk

RAIDs ‘ advantages :
- Performance , using multiple disks in parallel can greatly speed up I/O times
- Capacity , large data sets demand large disks
- Reliability , spreading data across multiple disks makes the data vulnerable to the loss of a single disk.

Interface : mirrored RAID system
Fault Model : Fail-stop model - working or failed . With a working disk, all blocks can be read or written. In contrast , when a disk has failed , we assume it is permanently lost.

RAID_0 :Striping(即将磁盘分为多个虚拟磁盘，同时每一行都算一个条带).
Larger chunk sizes reduce positioning time.

RAID_1:Mirroring.
With a mirrored system , we simply make more than one copy of each block in the system ; each copy should be placed on a separate disk , of course . By doing so , we can tolerate disk failures.

RAID_4:Saving Space with Parity(用异或思想计算奇偶校验位来压缩冗余信息)

P(new) = ( C(old) xor C(new) ) xor P(old)

RAID_5:Roatating Parity(即把每个奇偶校验码分布在每一个磁盘里)

#41 Fast File System - CRUX : How to organize on-disk data to improve performance

FFS by keeping the same interface to the file system but changing the internal implementation.

The Cylinder Group(即把整块磁盘的File System 分割成一个一个组)

FFS keeps a copy of the S in each group for reliability reasons.(每个组都有一个总的superlock)
A per-group ib and db serve this role for inodes and data blocks in each group.
The inode and data block regions are just like before

Allocate Files and Directories ‘ Policies : keep related stuff together

For directories : find the cylinder group with a low number of allocated directories and a high number of free inodes , and put the directory data and inode in that group.
For Files : first , it makes sure to allocate the data blocks of a file in the same group as its inode , thus preventing long seeks between inode and data . Second , it places all files that are in the same directory in the cylinder group of the directory they are in.

Large-File Exception : After some number of blocks are allocated into the first block group , FFS places the next “large” chunk of the file in another block group. Then , the next chunk of the file is placed in yet another different block group , and so on.

Innovation :
First , FFS solve internal fragmentation . Sub-blocks which are 512-byte little blocks that the file system could allocate to files. As the file grew , the file system will continue allocating 512-byte blocks to it until it acquires a full 4KB of data. (即先将512byte的小文件存起来，等到4kb时再一次性写进去）

Second , change disk layout to avoid full rotation read 1 after read 0.
简单来说，就是正常读完 0 之后，然后在准备读 1 的时候有可能指针已经已经过了 1 了 ，这样子可能要浪费时间转一圈再到 1 ，而更改布局可能会让刚好就读到 1 而不用转一圈)

FFS was the first file system to allow for long file names.

#42 FSCK and Journaling - CRUX : How to update the disk despite crashes

Crash scenarios :
- Just the data block is written to disk
- Just the updated inode is written to desk
- Just the updated bitmap is written to disk
- The inode and bitmap are written to disk , but not data
- The inode and the data block are written , but not the bitmap
- The bitmap and data block are written , but not the inode

Solution#1 : FSCK - File system checker
- Superblock : fsck first checks if the superblock looks reasonable
- Free blocks : next , fsck scans the inodes , indirect blocks , double indirect blocks , etc.
- Inode state : Each inode is checked for corruption or other problems.
- Inode links : fsck also verifies the link count of each allocated inode.
- Duplicates : fsck also checks for duplicate pointers
- Bad blocks : A check for bad block pointers is also performed while scanning through the list of all pointers.
- Directory checks : fsck does not understand the contents of user files; however, directories hold specifically formatted information created by the file system itself.

Fsck is too slow , it useful but wasteful.

Solution#2 : Journaling
When updating the disk , before overwriting the structures in place , first write down a little note describing what you are about to do .

By writing the note to disk , you are guaranteeing that if a crash takes places during the update of the structures you are updating , you can go back and look at the note you made and try again ; thus , you will know exactly what to fix after a crash , instead of having to scan the entire disk.

Data Journaling :

onsist of : TxB(头) , inode , bitmap , data block , TxE(尾)
Sequence of operations : 1.Journal write 2.Checkpoint (先写日志，如果没问题再更新数据）

Write Journal(five blocks write at once) is simple but slow . And need to improve

Problem : if disk loses power when write block , and maybe lose data block .

To avoid this problem , the file system issues the transactional write in two steps . Frist , it writes all blocks except the TxE block to the journal , issuing these writes all at once . When those complete , the file system issues the write of the TxE block , thus leaving the journal in this final , safe state.

Current protocol : 1.Journal write 2. Journal commit 3. Checkpoint(先写日志，然后确认日志是否合格，即判断是否丢失数据，如果丢失数据，不计入这一次日记，如果没丢失数据，则正常更新数据）

Journal log problems : When the log becomes full , two problems arises . First , the larger the log , the longer recovery will take . Second , when the log is full , no further transactions can be committed to the disk.

Circular log

Now four steps to journal protocol : 1. Journal write 2. Journal commit 3. Checkpoint 4. Free （多了一步释放步骤）

To reduce double-write cost , the data block , previously written to the log , would instead be written to the file system proper , avoiding the extra write.

New protocol : 1.Data write 2. Journal metadata write 3. Journal commit 4. Checkpoint metadata 5. Free

Solution #3 - Other Approach : 1. copy-on-write 2.backpointer-based consistency 3. optimistic crash consistency

#43 Log-structed File System - CRUX : How to make all writes sequential writes

Writing to disk sequentially is not enough to guarantee efficient writes.

Write buffering : before writing to the disk , LFS keeps track of updates in memory ; when it has received a sufficient number of updates , it writes them to disk all at once , thus ensuring efficient use of the disk .

Finding an inode given an inode number in LFS is difficult.
Inode map : The imap is a structure that takes an inode number as input and produces the disk address of the most recent version of the inode. Imap needs to be kept presistent , it could not live on a fixed part of the disk . Instead , LFS places chunks of the indoe map right next to where it is writing all of the other new information .

The Checkpoint Region : how we find the inode map ? LFS has just a fixed place on disk , known as the checkpoint region . It contains pointers to the latest pieces of the inode map , and thus the inode map pieces can be found by reading the CR first . Note the CR is only updated periodically ( say every 30 seconds or so) , and thus performance is not ill-affected.

Garbage Collection : LFS must periodically find these old dead versions of file data , inodes , and other structures , and clean them ; cleaning should thus make blocks on disk free again for use in subsequent writes .
The LFS cleaner works on a segment-by-segment basis , thus clearing up large chunks of space for subsequent writing .
Periodically , the LFS cleaner reads in a number of old segments , determines which blocks are live within these segments , and then write out a new set of segments with just the live blocks within them , freeing up the old ones for writing .

Now we are left with two problems . First , how can LFS tell which blocks within a segment are live , and which are dead ? Second , how often should the cleaner run , and which segments should it pick to clean ?

Determining Block Liveness : 对于在地址A的块D，先获取inode number 和 offset 然后在imap中查找 inode number 所在的位置并从磁盘中读取。最后，用offset来查看inode来看文件的offset算出来的块的位置。如果这个位置在地址A上，则segment是alive状态，否则是dead状态，这时候就可以回收。

Which Blocks to clean and when ? Either periodically , during idle time , or when you have to because the disk is full .
Segregate hot and cold segments . A hot segment is one in which the contents are being frequently over-written ; for such a segment, the best policy is to wait a long time before cleaning it, as more and more blocks are getting over-written (in new segments) and thus being freed for use . A cold segment, in contrast, may have a few dead blocks but the rest of its contents are relatively stable.

Crash recovery and the log

Write to the CR : 确保CR更新发生，在磁盘首尾各存有一个CR ， 交替更新它们。

Write to a segment : 因为LFS 30 秒写一次CR，file syste上一次一致的快照可能会很老。重启时可能会丢掉几秒的更新。利用回滚方法，从最后一个CR开始，找到Log，用日志来读取下一个segment并检查是否有有效更新，如果有的话，LFS会进行相应的更新同时恢复自上次CR写入的data和meadata

#44 Flash-based SSDs - CRUX : how to build a flash-based SSD

Flash chips level of storage : 1. single-level cell(SLC) , one single bit (1 or 0)
2.multi-level cell(MLC) , two bits (00,01,10,11)
3.triple-level cell(TLC) , three bits
Overall , SLC chips achieve higher performance and are more expensive .

Flash chips are organized into banks. A banks is accessed in two different sized units : blocks (size 128 KB or 256 KB) , pages ( a few KB in size , 4KB) . Within each bank there are a large number of blocks ; within each block , there are a large number of pages.

asic Flash Operations : 1. Read(a page) 2. Erase(a block) 3. Program(a page) 即先读page，然后再擦除page对应的block ，同时确保擦除前block中其他page被复制保存到其他地方去，最后再写入page

Flash Performance : Read is quick , however , write is expensive . To get good write performance , you will have to make use of multiple flash chips in parallel.
Flash Reliability : 1.wear out 2.disturbance , read disturbs or program disturbs.

Internal Logic : an SSD consists of some amount of flash chips , also contains some amount of volatile memory ; such memory is useful for caching and buffering of data . Contains control logic to orchestrate device operation .

One of the essential functions of this control logic is to satisfy client reads and writes, turning them into internal flash operations as need be. Flash translation layer(FTL) , takes read and write requests on logical blocks and turns them into low-level read , erase , and program commands on the underlying physical blocks and physical pages .
One key will be to utilize multiple flash chips in parallel . Another performance goal will be to reduce write amplification .

High reliability , to solve wear out . FTL should try to spread writes across the blocks of the flash as evenly as possible, ensuring that all of the blocks of the device wear out at roughly the same time; doing so is called wear leveling and is an essential part of any modern FTL.
To solve program disturbance . FTL will commonly program pages within an erased block in order, from low page to high page. This sequential-programming approach minimizes disturbance and is widely utilized.

Log-Structured FTL : Upon a write to logical block N , the device appends the write to the next free spot in the currently-being-written-to block ; we call this style of writing logging . To allow for subsequent reads of block N , the device keeps a mapping table ; this table stores the physical address of each logical block in the system . (即无脑按顺序写入，用表记录映射关系，尽管都对100进行修改，比如100 -> 0 之后再来个 100->4 ，也只是会在block1上的04page上写一个，同时映射也会改，但会带来垃圾回收的问题，毕竟block 0 的01，就没用了）


Garbage Collection - process : find a block that contains one or more garbage pages , read in the live pages from the block , write out those live pages to the log , and reclaim the entire block for use in writing .
As you see , garbage collection can be expensive , requiring reading and rewriting of live data . The ideal candidate for reclamation is a block that consists of only dead pages.

Mapping Table Size : The first cost is read , erase and write . And the second cost is large mapping tables.
One approach to reduce the costs of mapping is to only keep a pointer per block(instead of page)of the device . But small write cause significant performance problems.

Better Approach is hybrid mapping . With this , the FTL keeps a few blocks erased and directs all writes to them ; these are called log blocks .
The key to the hybrid mapping strategy is keeping the number of log blocks(i.e. page FTL) small . The FTL has to periodically examine log blocks and switch them into blocks that can be pointed to by only a single block pointer .
Three Condition : Switch merge , partial merge , full merge (from best to worst)

Some simple ways to reduce the memory load of the complexity of hybrid approach above , the simplest is just to cache only the active parts of the FTL in memory , thus reducing the amount of memory need .

Wear Leveling : the FTL must periodically read all the live data out of such blocks and re-write it elsewhere, thus making the block available for writing again.This process of wear leveling increases the write amplification of the SSD, and thus decreases performance as extra I/O is required to ensure that all blocks wear at roughly the same rate.

#45 Data Integrity and Protection - CRUX : How to ensure data integrity

Latent-sector errors(LSEs) : arise when a disk sector has been damaged in some way . For example , if the disk head touches the surface for some reason , it may damage the surface , making the bits unreadable .

Block corruption : a disk block becomes corrupt in a way not detectable by the disk itself . For example , buggy disk firmware may write a block to the wrong location ; in such a case , the disk ECC indicates the block contents are fine , but from the client’s perspective the wrong block is returned when subsequently accessed .
Similarly , a block may get corrupted when it is transferred from the host to the disk across a faulty bus ; the resulting corrupt data is stored by the disk , but it is not what the client desire.

Handling Latent Sector Errors : When a storage system tries to access a block, and the disk returns an error, the storage system should simply use whatever redundancy mechanism it has to return the correct data.

Checksum : a checksum is simply the result of a function that takes a chunk of data as input and computes a function over said data , producing a small summary of the contents of the data.

Checksum Functions :
- XOR : With XOR-based checksums, the checksum is computed by XOR’ing each chunk of the data block being checksummed, thus producing a single value that represents the XOR of the entire block.
Limitation : two bits in the same position within each checksummed unit change, the checksum will not detect the corruption.
- Addition
- Fletch checksum
- Cyclic redundancy check(CRC)

Stored checksum (read), computed checksum(compute) , if they equal , the data has likely not been corrupted , and thus can be safely returned to the user .If they do not match , this implies the data has changed since the time it was stored . In this case , we have a corruption , which our checksum has helped us to detect .

Problems :
- Misdirected writes : write the data to disk correctly but wrong location .
Use PID . For example, if the stored information now contains the checksum C(D) and both the disk and sector numbers of the block, it is easy for the client to determine whether the correct information resides within a particular locale .
- Lost writes : occurs when the device when the device informs the upper layer that a write has completed but in fact it never is persisted ; thus , what remains is the old contents of the block rather than the updated new contents .
Perform a write verify or read-after-write ; by immediately reading back the data after a write, a system can ensure that the data indeed reached the disk surface .

Overheads : space and time
- Space : 1. each stored checksum takes up room on the disk , which can no longer be used for user data . 2. When accessing data , there must now be room in memory for the checksums and then discards it one done .
- Time : 1. CPU must compute the checksum over each block , both when the data is stored and when it is accessed . One approach is to combine data copying and checksumming into one streamlined activity . 2. When checksums are stored distinctly from the data , and for any extra I/O needed for background scrubbing .
